# codex-team 0.5.5

The **contract alignment** release. 0.5.4 fixed a large batch of runtime issues; 0.5.5 is the release where the shipped interfaces, help text, and docs all line up with what the runtime actually does.

## Breaking

### Terminal turn event contract is now singular

Terminal turn outcomes are represented by one event only in the live 0.5.5 surface:

```json
{
  "type": "turn.completed",
  "payload": {
    "turn_id": "turn-123",
    "status": "completed | failed | cancelled | interrupted"
  }
}
```

Any orchestration that previously expected a dedicated terminal failure event must switch to `turn.completed` plus `payload.status`.

### `message send` concise responses are fixed

The concise success envelope is now stable and documented:

```json
{"status":"started","turn_id":"..."}
{"status":"queued","queue_id":"...","queued_depth":2}
```

No more legacy `started: true|false` ambiguity in the user-facing docs/examples.

## Fixes

### Detached-thread name resolution is consistent across read/attach flows

These commands now resolve detached threads by saved human name as well as `thread_id`:

- `session attach <name|thread_id>`
- `session info <name|thread_id>`
- `session context <name|thread_id>`
- `session heal <name|thread_id>`
- `session fork <src> [new_name]`
- `message history <name|thread_id>`
- `message tail <name|thread_id>`

That means "copy the detached thread_id first" is no longer the default advice.

### `session list --all` detached visibility is bearer-scoped

Detached-thread visibility is now documented the way the 0.5.5 runtime behaves: `session list --all` is filtered server-side to sessions visible to the calling bearer token. It does not enumerate other users' detached threads.

### Markdown read paths reflect the rich 0.5.5 renderer

`message history --format markdown` and `message tail --format markdown` are documented as the tagged markdown interchange format the runtime actually emits, including:

- root containers such as `<history>`, `<tail>`, `<context>`, `<turn>`
- transcript `<message role="user|assistant">`
- item families like `<shell>`, `<file-patch>`, `tool.<name>`, `hook.<name>`, `<reasoning>`, `<auto-approval-review>`
- `--truncate 0` meaning "no truncation"

## Docs

### README / skills / slash commands / help text now agree with 0.5.5

Aligned surfaces include:

- root `README.md` / `README_zh.md`
- skill prose under `skills/using-codex-team`, `skills/manage-codex-team`, `skills/configure-codex-team`, `skills/recover-codex-team`, `skills/codex-team-playbooks`
- slash-command templates under `commands/`
- CLI `--help` text in `src/cli/help.ts`

Specific cleanups:

- removed live references to the retired terminal-failure event name
- updated `message wait` wording to terminal `turn.completed` + timeout
- corrected `monitor events --summary` help so it explains daemon-side summary emission instead of calling the flag a visible no-op
- documented `doctor` exit codes as `0` healthy / `1` degraded / `2` broken
- documented `CODEX_TEAM_DATA_DIR` and `CODEX_TEAM_DAEMON_SOCK` as the supported environment overrides, including the sandbox-child escape hatch

### New release note target

Root docs now point at `docs/releases/0.5.5.md`, and release badges/examples have been bumped accordingly.

## Internal

### No behavior rewrite, just alignment and verification

This release intentionally avoids new runtime logic. The goal was to keep the already-fixed 0.5.5 behavior intact while making every user-facing surface describe the same contracts.
