# codex-team 0.5.4

The **legibility** release. Motivated by dogfood v3 feedback on 0.5.3: the diagnostic story is strong but still has gaps in the error-surface and discoverability layers. 0.5.4 closes those gaps.

## What's new

### Doctor — inline remediation hints

When `codex-team doctor` catches a problem, it now prints the fix alongside the verdict:

```
[FAIL] data_dir=/home/wzq/.codex-team not writable
       Try: CODEX_TEAM_DATA_DIR=/tmp/codex-team-1000 codex-team doctor
[FAIL] socket_bind EPERM — sandbox forbids listen()
       (no workaround — this environment cannot host the daemon)
[WARN] stale daemon.pid detected (pid 12345 not running)
       Hint: next `-b` call auto-cleans the pidfile and spawns a fresh daemon.
```

### Doctor — summary mode stays script-friendly

Programmatic consumers should use the human-readable default for diagnostics and `--short` for a one-line verdict snapshot:

```bash
codex-team doctor --short
# doctor=HEALTHY failed=none warned=none
```

### Doctor — plugin-mode launcher detection

When `CLAUDE_PLUGIN_ROOT` is set and the running launcher lives inside the bundled `bin/` path, the launcher check emits `[OK] launcher=<path> (plugin mode)` instead of the `[WARN] not on PATH` line. Cuts noise for the plugin-driven common case without hiding real misconfiguration.

### Daemon bootstrap — `data_dir_not_writable`

When the daemon can't spawn because the data dir isn't writable (EROFS / EACCES), you get a specific error code with structured data:

```json
{"ok":false,"error":{"code":"data_dir_not_writable","message":"daemon data_dir is not writable: /home/wzq/.codex-team (EROFS)","data":{
  "data_dir":"/home/wzq/.codex-team",
  "errno":"EROFS",
  "suggested_action":"set CODEX_TEAM_DATA_DIR to a writable path and retry, e.g. CODEX_TEAM_DATA_DIR=/tmp/codex-team-$USER. Run `codex-team doctor` to verify."
}}}
```

Previously this leaked as `daemon_unreachable` with a thin message, forcing users to reason about EROFS themselves.

### Daemon bootstrap — every error now points at doctor

Unclassified `daemon_unreachable` envelopes carry `data.suggested_action: "run \`codex-team doctor\` to diagnose"`. No more dead-end failure modes.

### CLI-side `--cwd` preflight

`session new`, `session fork`, and `session heal` now check `--cwd` before contacting the daemon. A nonexistent path or a file-not-directory fails with `invalid_params` immediately, even when daemon bootstrap is broken for an unrelated reason. The daemon-side preflight from 0.5.3 stays as defense-in-depth.

### `codex-team profiles` — bundled profile discovery

The 5 canonical role profiles are now queryable from the CLI (no more `grep` through skill docs):

```bash
# list all with their full session new flag bundles
codex-team profiles list
codex-team profiles list --short   # tab-aligned human table

# show one, with a copy-ready command
codex-team profiles show fixer
# {"name":"fixer","flags":{...},"command":"codex-team -b $TOK session new <name> --cwd <repo> --model gpt-5.4 --sandbox workspace-write --approval on-request --effort high --auto-approve '...'"}

codex-team profiles show fixer --short   # just the one-line command
```

No bearer token required. No daemon RPC. Single source of truth in `src/profiles/builtin.ts`.

## Stale-pidfile auto-cleanup (dogfood v2 → landed mid-0.5.3, noted here)

After `kill -9` of the daemon, the next `-b` call now auto-unlinks the dead pidfile + sock and spawns a fresh daemon (with a `[codex-team] auto-cleanup: ...` stderr notice). Previously required a manual `rm`.

## Migration

```bash
cd plugins/codex-team
npm install
npm run build
codex-team daemon restart
codex-team version    # cli_version: 0.5.4
codex-team doctor     # verify
```

No breaking changes in the session/daemon model. Current mainline CLI output now defaults to concise JSONL and pretty-prints `--full`; see the root README and CLI help for the latest shape.
