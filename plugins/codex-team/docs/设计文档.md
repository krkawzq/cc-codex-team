# codex-team 设计文档

版本：`codex-team 0.5.1`

codex-team 是 cli + daemon 工具，为 agent 提供 codex 的会话管理与事件订阅能力。

## 架构

### 工作原理

- `codex-team` cli：统一入口，类比 REST API 的 endpoint
- `codex-team-daemon`：后端服务，处理所有 cli 请求
- cli ↔ daemon 通过 Unix socket 通信（默认 `~/.codex-team/daemon.sock`）
- daemon ↔ codex 通过 codex app-server（官方 JSON-RPC）通信
- bearer token 做身份识别（类比 HTTP bearer auth），不同 user 数据隔离
- codex-team 只能调用 codex 暴露的接口，无法干涉 codex agent 本身的行为

### 核心概念

**session**：codex 内部管理的一轮完整对话上下文，有唯一 thread_id，由 codex 侧持久化。codex-team 不直接管理，**不能删除**。

**live session**：被 daemon 加载到 app-server 进程、可交互的 session。非 live 的 session 在 codex 侧仍然存在，只是 codex-team 不再跟踪。
- `attach`：将一个 session 变为 live
- `detach`：取消 live 状态（中断当前 turn + 从 app-server 卸载 + 从 codex-team 记录中移除）

attach/detach 是 codex-team 层的抽象，不要与 codex cli / codex app-server 自身的概念混淆。

**events**：codex 触发的事件。包括 session 级（turn 生命周期、approval 请求、askuserquestion 等）、系统级（网络错误、rate limit 等），以及高频流式 delta（token 级，默认过滤）。完整分类与字段见"事件模型"章节。

**monitor**：将 events 以 stdout 流形式推送给 agent。events list 在 daemon 内部**事件驱动**维护；monitor 默认**轮询式**批量推送（便于 agent 一次性处理），也支持 `--stream` 实时模式。

app-server 原生协议的详细映射见 `codex-app-server-protocol.md`。

## daemon

### 生命周期

- 单例绑定 sock 文件。每个 OS 用户 home 不同，sock 自然不同
- cli 调用时若未运行则自动 spawn
- 空闲自动关闭：仅在 **0 live session** 时才评估 idle；最近 daemon activity 超过阈值（默认 6h）时关闭。live session 本身视为 activity，因此后台仍有 live session 或持续 codex 通知时不会被 idle timer 关停；关闭会终止所有 codex 进程，不等待 pending turn
- 状态持久化；重启时自动恢复（含 live session 列表）

### app-server 管理

- 默认一个 app-server 进程承载多个 session（1:N 多路复用，上限由 `app_server.max_sessions_per_process` 决定）
- 不同 user 的 session 在不同 app-server 进程中隔离
- live session 与 app-server 进程不强耦合：daemon 重启、app-server crash 等情况下会按需重连 / `thread/resume`；当前版本**未实现**基于 `app_server.idle_unload_minutes` 的自动卸载
- **live session 是逻辑概念，底层进程管理是 daemon 黑箱**

## 事件模型

codex-team 事件源自 codex app-server 的 JSON-RPC notification 和 server-initiated request，经 daemon 规范化后暴露给 agent。原生协议见 `codex-app-server-protocol.md`。

**事件只含摘要**（`type` + 关键标识 + 状态 + 少量摘要字段），不包含完整 response body / 命令输出 / 文件 diff / 最终 turn 对象等详细内容。agent 需要详细内容时应调 `message tail` / `message history` / `session context`（带 `--format markdown` 可直接拿结构化文本，见 `html-md-format.md`）。

### 通用字段

所有事件共享：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | daemon 内单调递增的事件 id（用于 `--since` 断点续读） |
| `ts` | string (ISO8601) | 事件到达 daemon 的时间戳 |
| `type` | string | 事件类型（见下） |
| `session` | string \| null | session name；无 session 上下文的事件为 null |
| `thread_id` | string \| null | codex thread_id；无 thread 上下文的事件为 null |
| `payload` | object | 事件类型相关字段（大体透传自 app-server 原生 params） |

### session 事件

**turn / item 生命周期**

| type | 说明 |
|---|---|
| `turn.started` / `turn.completed` | turn 开始 / 结束。`turn.completed` 是**权威结束信号**；payload 摘要：`turn_id` / `status` / `duration_ms` / `item_count` / `token_usage`（不含 turn 内容） |
| `turn.error` | turn 内错误，`payload.will_retry` 表示 daemon 是否自动重试，`payload.error.codex_error_info` 是细粒度错误类型 |
| `turn.queued_started` / `turn.queued_failed` | 队列中的下一条 prompt 被自动派发 / 派发失败。`queued_failed` 的 payload 摘要：`queue_id`、`error.message`；失败项保留在队列中等待后续处理 |
| `item.started` / `item.completed` | item 粒度（assistant 消息、命令、文件变更、tool call 等）。payload 摘要：`item_id` / `type` / `status`（不含 item 内容） |

**thread 生命周期**

| type | 说明 |
|---|---|
| `thread.started` | thread 创建 |
| `thread.closed` | thread **永久关闭**，该 session 不可再用（codex-team 会自动 detach） |
| `thread.status_changed` | thread 状态变化 |
| `thread.token_usage_updated` | token 用量更新 |
| `thread.name_updated` / `thread.archived` / `thread.unarchived` | 元数据变化 |

**codex-team 内部事件**

| type | 说明 |
|---|---|
| `session.seized` | 本 user 下的 live session 被其它 user 用 `session attach --takeover` 抢走 |

**其它 session 事件**

| type | 说明 |
|---|---|
| `model_rerouted` | 模型因 rate-limit 被降级 |
| `context_compacted` | 上下文被 compact |
| `server_request_resolved` | 某个 approval / user_input 请求已被其它 client 回复，不需再回 |

### approval / user_input 请求事件

需要 agent 通过命令显式回复：

| type | 回复命令 | 说明 |
|---|---|---|
| `approval.command_execution` | `message approval` | 命令执行审批 |
| `approval.file_change` | `message approval` | 文件变更审批 |
| `approval.permissions` | `message approval` | 权限升级请求 |
| `approval.mcp_elicitation` | `message approval` | MCP 服务器请求结构化输入（form / url 两种 mode） |
| `user_input.request` | `message answer` | 工具向用户提问（askUserQuestion 语义） |

**共同字段**：`payload.request_id`（回复时必须带上）、`payload.kind`（approval 的子类型标识）、`payload.raw`（原生 params）。

超时未回复 / 被其它 client 抢答 → 对应的 `server_request_resolved`。

### 系统事件

这类事件多数是 daemon / codex 的全局通知，通常 `session` / `thread_id` 为 null；少数 daemon 自产事件会保留 session 过滤上下文：

| type | 说明 |
|---|---|
| `warning` | 通用提示；approval / user_input 回复耐久性告警也复用此 type，`payload.kind` 可为 `approval_reply_delivery_failed` / `approval_reply_backpressured` / `user_input_reply_delivery_failed` / `user_input_reply_backpressured` |
| `error` / `config_warning` / `deprecation_notice` | 通用错误 / 配置提示 / 弃用提示 |
| `account.updated` / `account.rate_limits_updated` / `account.login_completed` | 账号状态 |
| `mcp_server.status_updated` / `mcp_server.oauth_login_completed` | MCP 服务状态 |
| `fs.changed` / `skills.changed` / `app.list_updated` | 文件系统 / 技能 / app 列表变化 |
| `monitor.overflow` | `monitor events --interval` 的内部队列超过上限时发出；payload 含 `dropped_count` / `dropped_bytes` / `limit_events` / `limit_bytes` |

### 高频流式 delta（默认过滤）

这些事件 token 级发射，默认被 `monitor events` 过滤掉；agent 需显式用 `--include-delta` 订阅。

| type | 说明 |
|---|---|
| `item.agent_message_delta` | assistant 消息 token 流 |
| `item.command_exec_output_delta` | 命令 stdout/stderr 流 |
| `item.file_change_output_delta` | 文件变更预览流 |
| `item.reasoning_text_delta` / `item.reasoning_summary_text_delta` | 推理 token 流 |
| `item.plan_delta` | plan 项流 |

### 事件持久化

daemon 为每个 user 维护 append-only 事件日志，默认保留最近 `monitor.event_log_retention` 条（默认 10000）。`monitor events --since <id>` 可从任意未 rotate 的位置续读；id 已被 rotate 则报 `id_rotated` 并带上 `oldest_available_id`；id 从未存在（且不是 rotate 场景）则报 `invalid_params`。

## 错误模型

### CLI 错误信封

所有命令失败时输出：

```json
{
  "ok": false,
  "error": {
    "code": "error_code_str",
    "message": "human readable",
    "data": { ... }
  }
}
```

### 错误 code

| 分类 | code | 说明 |
|---|---|---|
| daemon | `daemon_unreachable` | sock 连不上（含 daemon 刚启动未 ready 的超时） |
| daemon | `internal` | daemon 内部 bug |
| user | `user_not_found` | 指定 token 未注册 |
| user | `user_already_exists` | `user create` 重复 |
| session | `session_not_found` | 当前 user 下没有该 session |
| session | `session_not_live` | 命令要求 live 但目标未 live |
| session | `session_busy` | `attach` 目标正在其它 user 下 live，需用 `--takeover` |
| request | `invalid_params` | 参数 / flag 错误、互斥 flag 同时给 |
| request | `invalid_decision` | approval 的 decision 与 request 类型不匹配 |
| request | `id_rotated` | `--since <id>` 指向的事件已被环形日志 rotate 掉 |
| codex | `codex_error` | app-server 返回错误（透传） |

### `codex_error` 数据结构

`error.data` 承载 app-server 原始错误：

```json
{
  "rpc_code": -32602,
  "rpc_message": "invalid params",
  "codex_error_info": "context_window_exceeded",
  "additional_details": null
}
```

**JSON-RPC code**：

| code | 含义 |
|---|---|
| -32700 | parse error |
| -32600 | invalid request |
| -32601 | method not found |
| -32602 | invalid params |
| -32603 | internal error |
| -32099 ~ -32000 | 服务器自定义（含 server_overloaded） |

**`codex_error_info` 枚举**（来自 `CodexErrorInfo`）：

| 值 | 说明 |
|---|---|
| `context_window_exceeded` | 上下文超限 |
| `usage_limit_exceeded` | 账号额度耗尽 |
| `server_overloaded` | 后端过载（daemon 会自动重试） |
| `http_connection_failed` / `response_stream_connection_failed` / `response_stream_disconnected` | 网络层（含 `http_status_code?`） |
| `response_too_many_failed_attempts` | 重试预算耗尽 |
| `internal_server_error` / `unauthorized` / `bad_request` | 基础错误 |
| `thread_rollback_failed` | thread rollback 失败 |
| `sandbox_error` | sandbox 内部错误 |
| `active_turn_not_steerable` | 当前 turn 拒绝 steer/interrupt（常见于 review / compact turn） |
| `other` | 未分类 |

### daemon 重试策略

daemon 对 `server_overloaded` 自动指数退避重试：

- 初始 `retry.initial_delay_seconds`（默认 0.25s），上限 `retry.max_delay_seconds`（默认 2s），最多 `retry.max_attempts`（默认 3 次），±20% jitter
- 重试耗尽后以 `codex_error` + `codex_error_info: response_too_many_failed_attempts` 上报

其它 codex 错误一律不重试，直接上报。

## 命令

### 全局

```
codex-team -b <token> <command> [args] [flags]
```

| flag | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `-b, --bearer <token>` | string | 是（除 `daemon` 命令组） | — | user 身份 token，agent 自选字符串 |
| `-v, --verbose` | bool | 否 | false | cli 自身 debug 日志写到 stderr，不影响 stdout JSON |
| `--daemon-sock <path>` | path | 否 | config 值 | 覆盖 `daemon.sock_path`，便于调试 / 隔离测试 |
| `-h, --help` | bool | 否 | false | 帮助 |

**顶层辅助命令**：

- `codex-team version`：输出 cli + daemon 版本（daemon 未启动只返回 cli 版本）。无 `-b`。

`--help` 是解析终止符：一旦命中，就按当前已匹配到的命令路径渲染对应 help 页面，后续 token 不再继续下钻。例如 `codex-team daemon --help user create` 显示的是 `daemon` help，而不是 `daemon user create`。各子命令均有独立 help 页（如 `codex-team session new --help`、`codex-team daemon config set --help`）。

### 输出

所有命令统一 JSON：

```json
{ "ok": true, "data": { ... } }
```

```json
{ "ok": false, "error": { "code": "...", "message": "...", "data": { ... } } }
```

错误 code 定义见"错误模型"章节。

## `codex-team daemon`

daemon 级命令，**不需要 `-b`**。

### `daemon status`

无输入，无 flags。输出 daemon 自身状态：pid、版本、uptime、sock 路径、data_dir、user 总数、app-server 进程数、资源占用摘要。

### `daemon start`

无输入，无 flags。显式 spawn daemon（cli 调用时通常自动 spawn，此命令用于脚本场景）。已运行 → idempotent noop。

### `daemon stop`

优雅关闭 daemon。会 detach 所有 live session、终止所有 app-server 子进程、持久化状态。

**Flags**

| flag | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `--force` | bool | 否 | false | 等价 SIGKILL，跳过 detach / 持久化。仅调试场景用 |

### `daemon restart`

等价 `daemon stop` 后再 `daemon start`。

### `daemon logs`

查看 daemon 日志文件（`daemon.log_path`）。

**Flags**

| flag | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `-f, --follow` | bool | 否 | false | tail -f 风格，新日志追加输出 |
| `-n <count>` | int | 否 | 100 | 先输出最后 N 行 |
| `--level <lvl>` | enum | 否 | — | 过滤 `error` / `warn` / `info` / `debug` / `trace` |

### `daemon user create <token>`

**输入**

| 参数 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `<token>` | string | 是 | — | 新 user 的 bearer token |

无 flags。token 重复 → `user_already_exists`。

### `daemon user destroy <token>`

**输入**

| 参数 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `<token>` | string | 是 | — | 待删除 user token |

**Flags**

| flag | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `--force` | bool | 否 | false | user 还有 live session 时仍然删除（默认存在 live session 会拒绝） |

默认模式下，若该 user 仍有 live session 则直接拒绝。带 `--force` 时关闭其 live session、取消 pending request，再清理 user 目录。

### `daemon user list`

无输入，无 flags。返回所有 user（token、创建时间、live session 数、最近活跃时间）。

### `daemon config get <key>`

**输入**

| 参数 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `<key>` | string | 是 | — | 配置键（点分层级，如 `daemon.idle_shutdown_hours`） |

无 flags。返回 `{ value, default, source: "explicit"\|"default", needs_restart: bool }`。

### `daemon config set <key> <value>`

**输入**

| 参数 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `<key>` | string | 是 | — | 配置键 |
| `<value>` | string | 是 | — | 值（按键类型解析） |

无 flags。冷键改动响应里带 `needs_restart: true`，需 `daemon restart` 才生效；热键立即生效。

### `daemon config unset <key>`

**输入**

| 参数 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `<key>` | string | 是 | — | 配置键 |

无 flags。恢复该键为默认值。

无位置参数。

**Flags**

| flag | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `--explicit-only` | bool | 否 | false | 只列 explicit 设置过的键 |

输出所有键（含默认值、当前值、冷/热、是否 explicit）。

### `daemon config reset`

**Flags**

## `codex-team session`

### `session new [name]`

**输入**

| 参数 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `[name]` | string | 否 | 自动生成（如 `s-${rand}`） | session 人类可读名 |

**Flags**

| flag | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `--model <m>` | string | 否 | `codex.default_model` | 模型名（如 `gpt-5.4`） |
| `--cwd <path>` | path | 否 | 当前目录 | codex 工作目录 |
| `--sandbox <mode>` | enum | 否 | `codex.default_sandbox` | `read-only` / `workspace-write` / `danger-full-access` |
| `--approval <policy>` | enum | 否 | `codex.default_approval` | `never` / `on-request` / `on-failure` / `untrusted` |
| `--effort <level>` | enum | 否 | `codex.default_effort` | `minimal` / `low` / `medium` / `high` / `xhigh` |
| `--personality <preset>` | string | 否 | — | personality preset |
| `--base-instructions <file>` | path | 否 | — | 系统级指令文件 |
| `--developer-instructions <file>` | path | 否 | — | 开发者级指令文件 |
| `--profile <name>` | string | 否 | — | codex config 中的预设 profile（同名字段被上述单独 flag 覆盖） |

创建成功后 session 自动 live。

### `session attach <name|thread_id>`

**输入**

| 参数 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `<name\|thread_id>` | string | 是 | — | session 标识 |

**Flags**

| flag | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `--takeover` | bool | 否 | false | 强行从其它 user 抢占 live 状态（默认会 `session_busy` 报错） |

行为：
- 可 attach 任意已存在的 codex session，无论是否本 user 创建
- session 已在其它 user 下 live：默认报 `session_busy`；带 `--takeover` 则先从原 user detach 再挂到本 user（原 user 收到 `session.seized` 事件）
- 本 user 下已 live：idempotent noop

### `message approval <name|thread_id> <request_id> [shortcut]`

回复 codex 的 approval 请求（`approval.command_execution` / `approval.file_change` / `approval.permissions` / `approval.mcp_elicitation`）。不同 approval 类型的 response 字段不同，命令同时支持**简易 shortcut** 和**完整 JSON**。

**输入**

| 参数 | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `<name\|thread_id>` | string | 是 | — | session |
| `<request_id>` | string | 是 | — | approval 请求 id（从 events 的 `payload.request_id` 获取） |
| `[shortcut]` | enum | 条件 | — | `accept` / `accept-session` / `decline` / `cancel`；与 `--json` / `--file` / `--stdin` 四选一 |

**Flags**

| flag | 类型 | 必填 | 默认 | 说明 |
|---|---|---|---|---|
| `--json <payload>` | string | 条件 | — | 完整 response JSON，用于复杂场景（execpolicy/network amendment、MCP form content、permissions scope） |
| `--file <path>` | path | 条件 | — | 从文件读 JSON |
| `--stdin` | bool | 条件 | false | 从 stdin 读 JSON |

**shortcut 语义**：

| approval 类型 | `accept` | `accept-session` | `decline` | `cancel` | 需 `--json` 的场景 |
|---|---|---|---|---|---|
| `command_execution` | `{decision:"accept"}` | `{decision:"acceptForSession"}` | `{decision:"decline"}` | `{decision:"cancel"}` | `acceptWithExecpolicyAmendment` / `applyNetworkPolicyAmendment` |
| `file_change` | 同上 | 同上 | 同上 | 同上 | — |
| `permissions` | 全部授权 + `scope: "turn"` | 全部授权 + `scope: "session"` | `scope: "turn"`，permissions 空 | — | 部分授权 / 自定义 scope |
| `mcp_elicitation` | `{action:"accept"}`（仅 url mode；form mode 需 `--json` 填 content） | — | `{action:"decline"}` | `{action:"cancel"}` | form mode 填 content |

约束补充：

- `approval.permissions` 不支持 `cancel`，使用会返回 `invalid_decision`
- `approval.mcp_elicitation` 不支持 `accept-session`
- `approval.mcp_elicitation` 的 form mode 不能走 shortcut `accept`，必须显式传 `--json` 并提供 `content`
- `message approval` / `message answer` 在 daemon 确认 app-server stdin 已接收后才返回；高背压时响应会略慢，并可能伴随 session 级 `warning` 事件

response JSON 结构见 `codex-app-server-protocol.md` 的 "Server-Initiated Requests" 段。

## 边界情况与恢复策略

### 生命周期与恢复

| 场景 | 处理 |
|---|---|
| daemon 崩溃，孤儿 codex 进程残留 | daemon 启动时读取持久化的 pid 清单，按 `pid + start_time + nonce` 身份追踪并回收登记过的 codex app-server；无法确认身份时保留到下次重试 |
| sock 文件残留（上次 daemon 异常退出） | 启动流程：先 `connect(2)` 探测 → 失败则读 pidfile → 检查 pid 是否仍存活且仍像是 `codex-team --daemon-internal` 进程；pid 死或 pid 已被其它程序复用 → 清理后重建；确认是活跃 daemon → 报错退出（双开保护） |
| app-server 子进程崩溃 | daemon 通过 stdio EOF 检测，自动 respawn + `thread/resume` 受影响 session；in-flight turn 丢失，推一条 `turn.error` + `codex_error_info: internal_server_error` |
| daemon 重启后的 live session | 持久化状态自动恢复；app-server 子进程不复活，第一次交互触发 lazy-spawn |
| daemon 空闲超时关闭 | 仅在 0 live session 且 idle 超阈值时触发；一旦关闭，当前 daemon 管理的 codex 子进程会被终止 |

### 并发与竞态

| 场景 | 处理 |
|---|---|
| 并发 attach 同一 session | 默认报 `session_busy`；`--takeover` 强行抢占（原 user 收到 `session.seized` 事件） |
| cli 之间并发写 registry | 所有注册表写操作走 daemon 单线程 handler，cli 间无文件级竞态 |
| daemon 双开 / pidfile 竞争 | 启动时除了检查 pid 存活，还会校验该 pid 仍是 codex-team daemon；活 pid 但身份不符不会阻塞启动 |
| detach 时有 pending approval / user_input | daemon 直接回 JSON-RPC 错误 `-32000 session detached` / `session seized by another user`；不会伪造成功响应 |
| `turn/steer` / `turn/interrupt` 遇 review/compact turn | app-server 返回 `active_turn_not_steerable`；daemon 透传为 `codex_error`，不重试 |
| 多 agent 对同一 live session 发 `send` | 按到达顺序入队，不做语义裁决（交由 codex / agent 自己处理） |

### 命名与身份

| 场景 | 处理 |
|---|---|
| session name 格式 | 命名规则 `^[A-Za-z0-9_\-]{1,128}$`；不允许 UUID 形态（`^[0-9a-f]{8}-...` 保留给 codex thread_id）和 `^th-` 前缀 |
| name ↔ thread_id 二义性 | `<name\|thread_id>` 先按 name 精确匹配 tenant 内的 session，失败再按 thread_id 匹配 |
| bearer token 含 FS 非法字符 | 持久化目录名使用 URL-safe base64 编码；原始 token 保存在目录里的 `metadata.json` |
| 两个 user 创建同名 session | 互不影响；name 只在单 user 内唯一 |

### 规模与资源

| 场景 | 处理 |
|---|---|
| 长 prompt / 大附件 | daemon 不设上限；由 codex 按 context window 拒绝（`context_window_exceeded` 事件） |
| 事件日志膨胀 | 按 `monitor.event_log_retention` 环形截断；`--since <id>` 遇已 rotate 返回 `id_rotated` + `data.oldest_available_id` |
| app-server 承载超 `max_sessions_per_process` | daemon 自动拉起新 app-server 进程；按 LRU 做 session 分配 |
| alarm 命令 runaway | `--timeout`（默认 60s）超时 SIGTERM → 5s 后 SIGKILL；事件写 `__alarm_event: timeout` |
| 单条事件 payload 过大 | daemon 不截断；agent 侧自己决定是否丢弃 |

### 版本与配置

| 场景 | 处理 |
|---|---|
| 跨版本升级 | 持久化文件带 `schema_version`；启动时低版本 → 执行 migration；高版本 → 拒绝启动（不自动降级） |
| config 冷键改动 | `config set` 响应里带 `needs_restart: true`；不 restart 则下次 daemon 启动时才生效 |
| daemon 刚启动 events 未 ready | `monitor events` 阻塞最多 5s 等待 ready；超时报 `daemon_unreachable` |
| config 值类型不匹配 | `config set` 报 `invalid_params` |

### 输入校验

| 场景 | 处理 |
|---|---|
| `--stream` + `--interval` 同时给 | `invalid_params` |
| `approval` shortcut 与 request 类型不匹配（如对 `user_input` 用 `accept`） | `invalid_decision` |
| `approval` / `answer` 同时给位置参数 + `--json` / `--file` / `--stdin` | `invalid_params` |
| `session new --profile` 与单独 flag 重叠 | 单独 flag 覆盖 profile 内同名字段；其余字段取自 profile |
| `--attach` 指向不存在的文件 | `invalid_params`，错误消息包含路径 |
| `--since <id>` id 不存在（未 rotate） | `invalid_params` |
